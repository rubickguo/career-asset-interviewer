# Career Web 测试用例

本文档记录当前本地网页闭环的基础测试范围。自动化脚本位置：

```bash
career-web/scripts/smoke-test-flow.mjs
```

运行方式：

```bash
cd /Users/yuxia/Documents/New\ project/career-web
npm run test:smoke
```

## 1. 环境与会话隔离

目标：确认测试不会污染真实用户数据。

- 使用独立 `X-Career-Session-Id`
- 工作目录必须落在 `workspace/sessions/<session-id>`
- 测试结束后清理临时 session
- 若设置 `KEEP_SMOKE_SESSION=1`，保留测试数据用于排查

## 2. 文案回归

目标：避免已知错误文案重新出现。

- 页面源码不应出现 `刚才这段里，最重要的可能是`
- 页面源码不应出现 `这些项目有没有能被别人相信的变化`
- 镜像卡标题应使用更自然的 C 端文案
- 项目深挖问题应指向“具体变化、指标、证据”，而不是抽象判断

## 3. Prompt 设计约束

目标：确认 AI 行为仍符合产品设计。

- 必须包含 `先理解人，再处理简历`
- 必须遵守三轮主流程：职业方向深访、项目证据提炼、简历策略与内容
- JD 必须是可选附加流程，不是主流程
- 个人网站必须是证据展示层，不是简历网页版
- 简历输出必须是正式可预览内容，而不是策略报告

## 4. 镜像卡反馈

目标：确认用户可以校准系统理解。

- 支持 `这很像我`
- 支持 `不完全是`
- 支持 `我想补充`
- `不完全是` 的具体原因会写入 session 内的反馈文件
- 后续 AI 调用可以读取这些反馈作为上下文

## 5. 简历渲染

目标：确认最终 HTML 简历不是 Markdown 报告。

- 必须生成 HTML 简历
- 必须生成检查报告 JSON
- HTML 中应包含姓名区、联系方式、定位句、关键词、个人简介、项目经历、个人作品、教育经历等结构
- HTML 中不应出现内部策略字段，例如项目排序、待确认问题、版式注意、风险提示、`layoutNotes`、`pendingQuestions`

## 6. 当前未覆盖

以下内容暂时需要人工测试：

- 真实 PDF / DOCX 上传解析质量
- 多轮 AI 回复的文案质感
- 移动端整页滚动与等待页体验
- PDF 导出后的分页、美观度和字号
- 真实 JD 分析分支
- 个人网站生成分支
